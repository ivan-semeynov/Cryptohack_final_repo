# Описание проведённых экспериментов

Этот файл содержит краткое описание экспериментов, выполненных в ходе работы над задачей, а также ссылки на Google Colab notebooks для удобного просмотра. Советую обратить внимание на написаный сервис предиктивной аналитики деградации катализатора в папке catalyst_risk_servvice.

В процессе работы команда разделилась на два направления:

1. **Baseline-решение**, ориентированное на получение устойчивого результата, соответствующего условиям соревнования.
2. **Экспериментальное решение**, в котором рассматривалась более сложная постановка:
   - многоклассовая классификация режимов работы;
   - регрессия времени до следующего режима / потенциального отказа.

## 1. Baseline-решение

### 1.1 Подготовка данных

#### Full Dataset Pipeline

Этот notebook собирает единый датасет из 6 независимых экспериментов по работе катализатора.

**Что делает pipeline:**

- автоматически находит папку проекта;
- загружает:
  - `data_i_WABT.csv`
  - `data_i_tech_params_long.csv`
  - `data_i_limit.csv`
- переводит технологические параметры из `long` в `wide` формат;
- синхронизирует данные на временные сетки:
  - `1h`
  - `3h`
  - `6h`
- выбирает основную рабочую сетку **3 часа**;
- приводит признаки к общим каноническим именам;
- объединяет составные признаки;
- добавляет инженерные признаки и лаги;
- собирает всё в единый `unified` датасет.

**Основной итоговый файл:**

`synced/unified_3h_full_pipeline_notebook.csv`

**Дополнительно сохраняются:**

- синхронизированные версии датасетов на `1h / 3h / 6h`;
- краткий summary по каждому датасету;
- статистики признаков в JSON.

**Какие признаки добавляются:**

Помимо исходных параметров, pipeline строит:

- `dataset_id`
- `limit_wabt`
- `gap_to_limit`
- `is_shutdown`
- `кратность_цвсг`
- `delta_t_отгона`
- лаги по ключевым технологическим признакам
- лаги и rolling-признаки по `WABT`
- `days_since_start`

**Почему это нужно:**

Такой pipeline приводит все 6 экспериментов к единой структуре и формирует датасет, готовый для:

- feature engineering;
- построения target;
- обучения ML-моделей;
- анализа деградации катализатора.

**Почему выбрана сетка 3 часа:**

Сетка `3h` выбрана как компромисс между:

- сохранением динамики процесса;
- снижением шума;
- уменьшением числа пропусков.

### 1.2 Обучение модели и интерпретация

#### Catalyst Risk Pipeline

`Catalyst Risk Pipeline` — это baseline notebook для Google Colab / Jupyter, который решает задачу прогноза риска роста `WABT` на горизонте.

**Что делает notebook:**

- находит папку проекта;
- загружает данные из `synced/unified_3h.csv`;
- строит честный target без утечки будущего;
- делит данные по времени на `train / validation / test`;
- обучает baseline-модель;
- считает метрики;
- сохраняет артефакты модели и отчёты.

**Постановка задачи:**

Предсказывается бинарное событие:

- `target_rise_in_horizon = 1`, если в ближайшие **30 дней** `WABT` вырастет минимум на **5°C** относительно текущего значения;
- иначе `0`.

Это задача **horizon classification**: по текущему состоянию установки оценить риск будущего роста температуры.

**Почему target честный:**

Target строится только из будущего окна внутри каждого `dataset_id`, а в признаки не попадают колонки с утечкой:

- текущий `WABT`;
- future-колонки;
- служебные target-поля.

Это делает эксперимент ближе к реальному прогнозу.

**Как делятся данные:**

Разбиение выполняется по времени внутри каждого `dataset_id`:

- `70%` → train
- `15%` → validation
- `15%` → test

Случайного перемешивания нет. Модель всегда обучается на прошлом и проверяется на будущем.

**Используемые модели:**

Изначально в качестве базовой модели использовался:

- `RandomForestClassifier`

Дополнительно baseline-решение было расширено тремя бустинговыми моделями, чтобы сравнить качество и устойчивость на той же постановке задачи:

- `XGBoost`
- `LightGBM`
- `CatBoost`

Таким образом, итоговое сравнение проводилось не только с одним baseline, а с набором сильных табличных моделей.

**Общие элементы обучения:**

- импутация пропусков медианой;
- балансировка классов через `class_weight` и/или `sample_weight`;
- подбор threshold по validation для части экспериментов;
- сравнение качества между моделями на одинаковом временном split.

**Какие метрики считаются:**

- `accuracy`
- `precision`
- `recall`
- `f1_score`
- `average_precision`

**Дополнительно:**

- сравнение с `DummyClassifier`;
- подбор threshold по validation;
- confusion matrix;
- permutation feature importance;
- сравнение `RandomForest` с `XGBoost`, `LightGBM` и `CatBoost`.

### 1.3 Проблемы, возникшие в baseline-решении

В ходе работы были выявлены несколько важных проблем:

- первоначальная оценка качества была завышена из-за некорректного разделения временных рядов;
- ошибку удалось обнаружить и исправить;
- изначально target строился с использованием признаков, содержащих информацию из будущего, в частности использовался `gap_to_limit`, что приводило к утечке. Поэтому решение подкреплено дополнительным sanity check, когда из выборки выкидываются все датасеты и остается один для обучения остальные для теста и валидации, так со всеми сетами. https://colab.research.google.com/drive/1JJ0ffTPvXZ_MyNVwKN7xr1w30T4xZVdI?usp=drive_link

После исправления этих ошибок baseline был приведён к более честной постановке.

## 2. Экспериментальное решение

В рамках дополнительного исследования была рассмотрена более сложная постановка задачи:

- многоклассовая классификация режимов работы;
- регрессия времени до следующего режима / потенциального отказа.

Рассматривались 4 класса:

- устойчивый режим эксплуатации;
- перегрузка;
- дефицит нагрузки;
- неявный режим работы.

---

### 2.1 Многоклассовая классификация следующего режима

В первом эксперименте была поставлена задача предсказывать **следующий режим работы** и время до него.

Однако удовлетворительных метрик качества получить не удалось. Основные причины:

- высокий уровень шума в данных;
- сложная временная структура;
- целевые метки формировались на основе робастных z-оценок и эвристик, что вносило дополнительную неопределённость.

### 2.3 Попытка переразметки режимов через кластеризацию

Далее была предпринята попытка заменить эвристическую разметку на кластерную.

Использовались алгоритмы кластеризации, однако оказалось, что модели лучше улавливают различия между отдельными экспериментами, чем общие режимные паттерны. Иными словами, кластеризация выделяла особенности конкретных датасетов, а не универсальные классы состояния.
